# 糖豆人记账游戏 (SugarRunner Expense) - 技术规格评审结果

> **评审日期**：2026-06-30  
> **评审范围**：spec.md、个人记账本项目需求搜集.md、checklist.md、tasks.md（对照参考 PRD.md、设计想法.md）  
> **评审人角色**：技术架构师 + 产品经理

## 一、需求完整性 —— 遗漏的功能点

### 1.1 文档间存在矛盾，需统一口径

| 矛盾点 | 文档 A 说法 | 文档 B 说法 | 影响 |
|--------|------------|------------|------|
| **数据存储位置** | 设计想法.md："所有数据保存在浏览器的数据库中（IndexedDB）" | spec.md："Node.js + SQLite 后端服务" | **架构级矛盾**，需明确到底用纯前端还是全栈方案 |
| **右上角数字含义** | 设计想法.md："右上角数字表示年度花费" | spec.md：(未明确年度/月度语义) | 需确认余额是年度累计还是月度结余 |
| **统计粒度** | 设计想法.md："本周统计" | spec.md："月度收支汇总 `month=2026-06`" | 需统一是按周还是按月统计（或两者都支持） |
| **消费预警/预算** | PRD.md 项目目标："通过预算设置帮助用户控制开支" | spec.md：无任何预算/预警功能设计 | 预算功能在 PRD 标注为核心价值但完全未设计 |
| **金额输入方式** | PRD.md："快捷金额行（10/20/50/100/200）" | 设计想法.md：仅提"快捷数字按钮" | 无自定义金额输入能力，只能选固定 5 个值 |

### 1.2 需求收集中提及但规格中缺失的功能

| 缺失功能 | 来源 | 建议 |
|----------|------|------|
| **消费预警/预算设置** | 个人记账本项目需求搜集.md 核心价值："消费预警"；PRD.md 项目目标第3条 | 补充月度预算设置、超支提醒功能 |
| **账单备注搜索/筛选** | 通用记账需求 | bills 表虽有 `note` 字段，但 API 不支持按备注搜索 |
| **自定义金额输入** | 设计想法.md 隐含 | 配置面板仅支持固定金额按钮，应增加自定义金额输入框 |
| **数据导入** | 对标导出功能 | 只有导出无导入，用户无法从其他记账软件迁移数据 |
| **分类图标自定义** | 通用需求 | PRD 提到"设置名称和图标"，但 spec 中分类创建仅传 `name` 和 `icon`(emoji)，无图标上传/选择机制 |
| **记账类型切换（收入/支出）** | bills 表有 `type` 字段 | 游戏配置面板中如何选择"收入"还是"支出"未设计，预设分类中"收入"是分类而非类型 |

### 1.3 其他缺失项

| 缺失项 | 说明 |
|--------|------|
| **撤销最近一笔记账** | 用户误操作后无恢复手段 |
| **离线记账能力** | 网络断开只能提示失败，不支持离线暂存+恢复联网后同步 |
| **账单单条备注查看** | 账单列表只显示金额+分类+时间，备注需点开详情（未设计详情页） |
| **多端同步** | 用户故事隐含云端需求，但单机 SQLite 无同步机制 |

## 二、边界条件 —— 未处理的异常情况

### 2.1 服务端边界

| 异常场景 | 现状 | 风险等级 | 建议 |
|----------|------|----------|------|
| **服务重启后验证码丢失** | 验证码存于 Node.js 进程内存 | 🔴 高 | 考虑 Redis 或至少提示用户服务重启需重新获取 |
| **QQ SMTP 不可用** | 无降级方案 | 🔴 高 | 增加 SMTP 连接检测告警 + 日志记录；考虑备用邮件服务 |
| **SQLite 并发写入冲突** | SQLite 单写者限制 | 🟡 中 | 高并发场景下 WAL 模式可缓解，但需测试确认 |
| **better-sqlite3 编译失败** | 原生模块依赖 Python + C++ 编译工具链 | 🟡 中 | 在 tasks.md / README 中注明 Node.js 版本要求和编译前提 |
| **大额 CSV 导出内存溢出** | 无分页/流式导出 | 🟡 中 | 账单超 10 万条时全量导出可能 OOM，建议流式写入 |
| **JWT 签名密钥泄露后无轮换机制** | 无 token 版本/失效机制 | 🟡 中 | 建议增加 JWT jti + 服务端黑名单 |
| **验证码锁定状态无持久化** | 锁定存内存，重启后锁定解除 | 🟡 中 | 验证码爆破锁定需持久化 |

### 2.2 前端边界

| 异常场景 | 现状 | 风险等级 | 建议 |
|----------|------|----------|------|
| **Canvas resize 时动画进行中** | 仅说了"resize 时重新计算" | 🟡 中 | 需定义 resize 时动画小球坐标/粒子状态如何处理 |
| **浏览器不支持 Canvas** | 未检测 | 🟢 低 | 建议增加 `canvas` 能力检测 + 降级提示 |
| **多次快速点击同一小球** | 未定义防抖策略 | 🟡 中 | 配置面板已打开时再次点击需做防重处理 |
| **requestAnimationFrame 在后台标签页暂停** | BC-16 提到"暂停移动" | 🟢 低 | 已处理，但需注意 `deltaTime` 计算避免时间跳跃 |
| **localStorage 满** | 未处理 | 🟢 低 | JWT 存储极小，风险低，但 load/save 需 try-catch |
| **emoji 渲染不一致** | 分类图标用 emoji，不同 OS 显示效果不同 | 🟢 低 | 建议在兼容性说明中注明 |
| **小球全部被吃掉且生成位置与糖豆人重叠** | 未定义 | 🟢 低 | 建议生成时避开糖豆人当前位置 |

### 2.3 校验与边界

| 缺失项 | 说明 |
|--------|------|
| **金额上限校验** | 无最大金额限制，理论上可输入极大值导致统计溢出 |
| **自定义分类名长度限制** | 未定义，过长名称可能破坏布局 |
| **账单备注长度限制** | 未定义，前端列表展示可能被撑破 |
| **邮箱唯一性竞态条件** | 验证码登录"自动创建"时，并发请求可能创建重复账号 |

## 三、技术可行性 —— 实现难点

### 3.1 高风险技术点

| 难点 | 详细说明 | 建议 |
|------|----------|------|
| **Canvas + DOM 点击事件协调** | 小球/糖豆人在 Canvas 层绘制，配置面板/侧边面板是 DOM 层。Canvas 上无法直接绑定 DOM 事件，需手动做坐标命中检测（hitTest），且需处理事件穿透问题 | 建立统一的事件管理器，Canvas 层用全局 click 事件 + 手动坐标计算，DOM 面板层需 `e.stopPropagation()` 防止穿透 |
| **贝塞尔曲线小球飞行动画** | 小球从任意位置飞向移动中的糖豆人嘴巴，需要实时计算终点位置和曲线控制点 | 建议在糖豆人到达小球位置后先暂停移动，再播放飞行动画（已在 spec 中定义到达→进食状态），降低复杂度 |
| **糖豆人 Canvas 2D 手绘** | 纯 Canvas 2D 手绘糖豆人（身体、眼睛、嘴巴、短腿、表情变化），无任何图片素材 | 实现复杂度较高，建议分阶段：先用简单几何图形验证流程，再细化外观 |
| **星星粒子池复用** | spec 提到"使用粒子池复用，避免 GC"，实现粒子对象池需要小心管理活跃/空闲状态 | 建议使用固定大小的数组 + 标记位管理，而非频繁 new/delete |
| **Canvas 饼图手绘** | 含 hover 高亮、标签定位、百分比计算等交互，纯 Canvas 手写比用 ECharts 复杂很多 | 评估是否值得节省 ECharts 的引入成本；若坚持手绘，先写独立 PieChart 类 |
| **全屏 Canvas 自适应 resize** | `window.innerWidth × window.innerHeight` resize 时需重算所有小球坐标、糖豆人位置、HUD 位置，且动画状态不能中断 | resize 建议做 debounce（200ms），resize 完成后一次性重算，动画中标记 `needsRelayout` |

### 3.2 中风险技术点

| 难点 | 说明 |
|------|------|
| **验证码登录自动创建账号** | 需在事务中原子完成"查用户→不存在则创建→签发JWT"，避免并发竞态 |
| **SQLite 日期查询** | bills 表 `created_at` 存 TEXT 格式，日期范围查询需确保格式化一致 |
| **余额数字滚动动画** | 需用 `requestAnimationFrame` 做数字递增动画，注意精度 |
| **CSV 中文/emoji 编码** | 需加上 BOM (`\uFEFF`) 确保 Excel 正确识别 UTF-8 |

### 3.3 架构层面待确认

| 问题 | 说明 |
|------|------|
| **前后端分离 or Express 托管静态文件？** | spec 说"Express 托管 client/ 目录"，这意味着前后端同域部署，CORS 配置可简化，但也限制了前后端独立部署/扩展 |
| **WebSocket 是否需要？** | 当前纯 HTTP 请求，账单数据不会有实时多人协作场景，不需要 WebSocket。但余额更新需前端在 API 响应后本地更新 |

## 四、安全风险 —— 安全漏洞分析

### 4.1 高危风险

| 风险 | 说明 | 修复建议 |
|------|------|----------|
| **JWT 存 localStorage** | localStorage 可被 XSS 攻击读取，token 泄露后攻击者可冒充用户 | 考虑 httpOnly cookie 或 access token + refresh token 模式；至少加强 XSS 防护（CSP header） |
| **密码强度仅 6 位** | "密码至少 6 位"无复杂度要求，纯数字 6 位可暴力破解 | 建议至少 8 位，包含字母+数字 |
| **无 CSP (Content-Security-Policy)** | 前端无构建工具、手写 JS，若存在 DOM XSS 无 CSP 兜底 | 建议在 Express 中配置 CSP header |
| **未提及 HTTPS** | spec 中 API base URL 为 `http://localhost:3000`，生产环境必须 HTTPS | 在部署文档中明确要求生产环境使用 HTTPS（Nginx 反向代理 + Let's Encrypt） |
| **JWT_SECRET 默认值** | spec 注释"默认值仅开发用"——极易部署时遗忘 | 启动时检测是否为默认值，若是则拒绝启动（生产环境） |

### 4.2 中危风险

| 风险 | 说明 | 修复建议 |
|------|------|----------|
| **无 CSRF 防护** | 基于 JWT Bearer Token 的 API 天然防 CSRF（不依赖 cookie），但仍应确认 | 若后续加入 cookie 认证则必须加 CSRF token |
| **验证码频率限制不均衡** | spec 说"单 IP 单日上限 20 次"，但"单邮箱 5 次错误后锁定 15 分钟"——IP 限制容易误伤（NAT 网络） | 建议以邮箱维度为主，IP 维度为辅（适当放宽 IP 限制） |
| **账单 API 无归属校验代码示例** | spec 描述"查询时强制带 `WHERE user_id = ?`"，但需确保每处实现都严格执行 | 建议封装统一的 `db.bills.findByUser(userId, ...)` 方法，避免遗漏 |
| **验证码无图形验证码** | 纯邮箱验证码接口可被脚本自动化调用 | 建议在发送验证码前增加图形验证码（或至少增加更严格的频率限制） |
| **`.env` 泄露风险** | spec 提到 SMTP 授权码存于 `.env`，已在 `.gitignore` 中，但需确保不会通过其他方式泄露（日志、错误消息等） | 后端错误响应中不应包含环境变量值；生产环境日志脱敏 |

### 4.3 低危风险

| 风险 | 说明 | 修复建议 |
|------|------|----------|
| **无请求体大小限制** | express.json() 默认无限制，可能被大 payload 攻击 | 添加 `express.json({ limit: '1mb' })` |
| **账号注销无冷静期** | 注销后数据不可恢复，可能被恶意/误操作利用 | 可考虑 7 天冷静期（软删除）|
| **Express 默认错误页暴露技术栈** | 生产环境 500 错误可能泄露堆栈信息 | 自定义错误处理中间件 |

## 五、用户体验 —— 改进建议

### 5.1 交互逻辑问题

| 问题 | 详细说明 | 建议 |
|------|----------|------|
| **"零文字引导"与新手体验冲突** | 设计想法强调"页面没有任何文字提示或弹窗说明"，但新用户首次进入游戏完全不知道要点击小球 | 首次进入增加 3 秒引导动画：糖豆人走到一个小球旁边"示范"点击→配置→吃掉，之后不再出现 |
| **仅 5 个固定金额不够灵活** | 用户想记录 15 元早餐或 3.5 元公交无法精确输入 | 增加自定义金额输入框，与快捷金额并存 |
| **无音效反馈** | 进食动画缺少音效会大大降低游戏感 | 增加小球飞行音效、爆炸音效、打嗝音效（使用 Web Audio API，可选） |
| **糖豆人移动不可控** | 糖豆人完全自动，用户可能觉得"被动" | 可考虑增加键盘方向键微调位置（不强制，作为隐藏特性） |
| **侧边面板 35% 宽可能太窄** | 在 1366px 宽度下约 478px，饼图+列表+时间轴挤在一起 | 建议面板宽度为 `min(480px, 40vw)` |
| **登录页与游戏页视觉风格未必连贯** | 登录页 DOM 表单 → 游戏页 Canvas 全屏，过渡生硬 | 登录成功后做一个糖豆人跳跃过渡动画进入游戏 |
| **无法记录"收入"** | 配置面板只有支出分类可选，预设的"收入"分类无法在游戏流程中使用 | 在小球配置面板增加"收入/支出"切换，或将收入作为独立交互（如金色特殊小球） |

### 5.2 视觉与可用性

| 改进点 | 建议 |
|--------|------|
| **色盲友好** | 6 种小球颜色中有红绿相近色，建议增加形状差异（如条纹、斑点）辅助区分 |
| **移动端适配** | PRD 提到"第二阶段移动端"，但 spec 完全按桌面端设计，Canvas 全屏 + 侧边面板在手机上体验会很差。建议尽早考虑响应式基准 |
| **余额数字对比参考** | 仅显示余额数字，用户无法感知是多是少。建议饼图旁边显示"比上月 +15%"等对比信息 |
| **快捷金额可配置** | 不同用户消费习惯不同（学生 10-50，白领 50-200），可允许用户在设置中自定义快捷金额列表 |
| **暗色模式** | 长期记账场景下（尤其夜间），暖色渐变可能刺眼。后期可考虑暗色主题切换 |

### 5.3 数据一致性体验

| 问题 | 说明 |
|------|------|
| **余额与账单列表不同步** | 进食完成后余额更新是乐观更新（前端立即改）还是等 API 返回？若是乐观更新，API 失败时需回滚 |
| **账单编辑后余额更新** | 编辑/删除账单后余额是否联动更新？checklist 有 V-40 验证，需确保实现 |
| **多 Tab 同时打开** | 同一浏览器多 Tab 打开同一账号，一侧记账后另一侧余额不会更新 |

## 六、总结与优先级建议

### 必须在开发前解决（阻塞级）

| 序号 | 问题 | 归类 |
|------|------|------|
| 1 | **明确纯前端 or 全栈架构**（设计想法 vs spec 矛盾） | 需求完整性 |

| 4 | **JWT 不要存 localStorage**，或至少配置 CSP header | 安全风险 |

> **评审结论**：整体设计思路完整、创新性强，游戏化记账的概念清晰。主要风险集中在文档间架构矛盾（纯前端 vs 全栈）、关键交互细节缺失（自定义金额、收入类型切换）和安全配置不足三个方面。建议在动工前优先解决 5 个阻塞级问题。
